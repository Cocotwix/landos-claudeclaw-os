# LandOS Known Limitations

Purpose: track intentionally unfinished work, why it is unfinished, ownership,
when to revisit it, and whether it blocks current business use.

Do not store secrets, private seller data, real APNs, private addresses, or raw
property work product here.

| Limitation | Why Unfinished | Owner / Department | Revisit When | Blocks Current Business Use |
|---|---|---|---|---|
| Dashboard route/UI changes need a rebuild + service restart to go live | The prod service runs the COMPILED build (`node dist\index.js`), not `tsx`, and there is no hot reload. New/changed routes (e.g. the Browser Training endpoints) return 404 on the running server until `npm run build` recompiles `dist` AND the service is restarted. This bit us as the training usage 404 (stale `dist` from Jul 5 02:13). Standing operational gotcha, not a code bug. | Ops / Dashboard | Whenever routes or server code change — always rebuild + restart before Operator QA | No (fixed for this change) — but any future backend change hits the same gap until built + restarted |
| Browser Training left a QA-test artifact in the live DB | The runtime-wiring verification created one ended test session + a `qa_runtime_wiring_test` DRAFT playbook in `store/landos.db`; usage now shows 1 lifetime session / 1 playbook. Drafts cannot execute, so it is harmless, but it slightly inflates the counters. There is no delete endpoint yet, so removing it needs a direct DB op (gated). | Browser Training | If Tyler wants a clean usage counter | No — cosmetic; the draft is inert |
| Trained-playbook field extraction needs selectors | RESOLVED 2026-07-05: training now captures per-field selectors. When Tyler names a field ("this is road frontage"), `captureFieldBinding` binds it to a stable selector (click-probe / data-testid / stable id) or a nearby-label anchor (fallback), stored in `landos_training_field_binding` and attached to the playbook `fieldSelectors` at synthesis. Live execution extracts (selector-first, label fallback) and writes to the Deal Card with confidence + provenance. Residual: binding quality depends on the page exposing a stable selector/label; brand-new/unusual layouts may bind label-only (low confidence). | Browser Training / Browser Agent | If a site exposes neither a stable selector nor a clean label | No — capture + extraction + writeback work; only selector robustness varies by site |
| Field binding needs a live authenticated page to reach the DOM | The high-confidence binding (real selector/id + sample value) and live extraction read the DOM via the operator's authenticated Chrome (`withWorkingPage`, gated by `BROWSER_INTEL_LIVE`). Without a live page, capture still stores a label-only binding (usable at runtime via label matching) but cannot confirm a selector or sample value. So voice-only training on a screen-shared foreign tab yields label bindings; full-strength bindings need the LandOS-driven Chrome. | Browser Training / Browser Session | When binding is driven against the operator's logged-in Chrome | Partially — label bindings work everywhere; selector-strength bindings need a live page |
| Live trained execution needs the operator's authenticated Chrome | `runTrainedPlaybook` live mode drives the shared CDP browser session (`withWorkingPage`), which is gated by `BROWSER_INTEL_LIVE` and requires the operator's logged-in LandPortal Chrome. In the headless CC environment `configured()` is false, so a live run returns an honest `not_configured` result (nothing fabricated). Dry-run against a reachable page works; full live LandPortal replay must be driven from Tyler's browser. | Browser Agent / Browser Session | When driven from the operator's own Chrome session | Partially — dry-run + unit-verified; true live replay pending an authenticated session |
| Property Board workspace is not yet operator-usable | UI/persistence wiring still needs to render persisted inspection, comps, market, and discovery output visibly | Acquisition Specialist / Property Board | Current sprint | Yes |
| Weak duplicate Property Cards may appear for same property | Earlier raw-intake runs created unverified duplicates before verified-card suppression was fully wired | Property Resolution / Property Board | Current sprint | Yes |
| Old Duke/LandPortal paid-credit UI may still appear in new Property Card flow | Legacy UI was not fully removed from Property Board | Property Board | Current sprint | Yes |
| Market Intelligence must remain reusable and separate | Capability exists structurally, but dashboard-visible integration is still being hardened | Market Intelligence | After Property Card workspace is usable | Partially |
| County Records fallback not fully validated live | Routing exists structurally; needs live failure-path acceptance without paid tools | Property Inspection / County Records Browser | After Property Card acceptance | No, unless LandPortal fails |
| Reference UI artifact library is newly formalized | Folders exist; real redacted screenshots must be added during Operator QA | Operator QA | Next dashboard QA run | No |
| Land Score depends on how complete the LandPortal read was | Mostly resolved 2026-07-04: the rubric now consumes the LandPortal parcel fact sheet (road frontage, wetlands, FEMA, buildability, acreage, valuation) + gov-DD cross-check, so a fully-read parcel scores fully (card #5 → 77/100 PURSUE). USGS 3DEP slope is now wired into Buildability, so a thin read that lacks LandPortal buildability still scores it from terrain (card #1 → 8/10). A thin read still gaps fields NO approved provider returned (e.g. FEMA unmapped) — honest, never fabricated. | Due Diligence / Land Score | When a parcel needs a fuller read | Minor — score is real and useful; only genuinely-absent fields gap |
| LandPortal authenticated browser flow not runnable in the CC environment | The provider steps — initial LandPortal page screenshot, reading sidebar/parcel-panel counts, clicking "show comps on map" and capturing it — need the operator's LIVE logged-in LandPortal (Chrome) session. The headless CC agent has no authenticated session, so it cannot perform those captures autonomously. The rich LandPortal parcel facts on the verified test card came from a PRIOR real operator browser read (persisted); a fresh headless lead cannot reproduce them. | Browser Agent / Property Resolution | When driven from the operator's own browser session | Partially — verified cards with a prior read are rich; brand-new headless leads are area-context until a real LandPortal read or a strong Realie identity |
| No 3D/terrain screenshot; Street View may not face the parcel | Google satellite + Street View are captured and displayed, but no 3D/terrain view is captured (the report itself flags "3D terrain screenshot still needs confirmation"), and the Street View heading is not yet aimed at the subject parcel centroid (can face down the road). | Visual capture (google-visual-capture) | Next visual sprint | No — satellite + a Street View render; refinement pending |
| Deal Card is discovery-ready but not offer-ready without sold comps | Paid comp credits are gated (LANDOS_ALLOW_PAID_COMPS off), so valuation shows asking-market $/acre only. A fresh lead produced 0 comps. Offer math needs sold comps + confirmed access/title/utilities. | Comparable Intelligence / Underwriting | When a sold-comp source is enabled/approved | Yes for offer-stage; no for discovery |
| Web frontend is not typechecked in the build | `vite build` uses esbuild (no tsc), and `web tsconfig` has PRE-EXISTING errors (AgentSuggestions.tsx, BrainGraph3D.tsx), so web tsc isn't enforced. That let `dashboardToken()` (a not-callable value) ship and crash the report block. Enforcing web tsc would catch this class of bug but first needs those pre-existing errors fixed. | Frontend / build | When web tsc is cleaned + wired into build/CI | No now (bug fixed) — but the gap that hid it remains |
| Property Board visual count is an aggregate, not a gallery | `workspace_visual_count` = current inspection assets (screenshots/overlays) + distinct persisted Google captures. It signals "how much visual context this property has," not a curated/deduped image gallery; the exact number can vary with what a given inspection persisted. Comp count and inspection presence are exact. | Acquisition / Property Board | If operators want a precise visual gallery count | No — it is an honest at-a-glance signal; open the card for the actual visuals |
| USGS slope is a coarse point estimate | The USGS 3DEP Buildability signal comes from a 33 m EPQS 5-point cross (a single average slope at the parcel point), not a full-parcel DEM slope distribution. It can over/under-state slope near band boundaries (e.g. card #5's 10.3% just crosses the 10% caution threshold). It is a cross-check, not a survey. | Due Diligence / Land Score | Deeper-DD terrain sprint | No — conflicts are surfaced; LandPortal remains the primary buildability source |
| Pre-existing failing test: property-card weak-duplicate merge | `src/landos/property-card.test.ts > "weak/address-only input does NOT merge into a verified card by address_key"` fails at commit 60c8378 (confirmed by stashing the DD-report changes). Unrelated to the DD-report/Land-Score/docs work; no change to property-card.ts/db.ts in these sprints. | Property Resolution / Property Board | Next Property Board sprint | No — isolated to the weak-duplicate-merge path |
